# Jotform Descriptions Import — Design

Date: 2026-05-19
Owner: alanhon1

## Problem

Every product in `data/products.json` carries four prose fields
(`description`, `indication`, `packaging`, `protocol`) that were
generated by an LLM during earlier scaffolding. They read as boilerplate
and don't reflect the actual product copy that the business publishes
on the public Jotform order page
(`https://form.jotform.com/shcoresteticsglobal/skin-global-product-order-form`).

The user wants the Jotform copy to become the source of truth wherever
it provides one, and to fill the rest from web research afterwards. The
`protocol` field in particular should end up much more specific than
the current generic "for professional use" stub.

## Source of truth

- **Phase 1**: `scripts/jotform-raw.html` (already fetched yesterday;
  re-fetched and diffed at script start, replaced if changed).
- **Phase 2** (later): generic web research per product.
- **Phase 3** (later): category-template-based protocol expansion
  (preferred) plus optional per-product web lookup.

## Non-goals

- No schema change. Reuse `description`, `indication`, `packaging`,
  `protocol`.
- No translation work — only the English source rows. `data/translations/`
  is out of scope.
- No price / image changes here.
- No DOM scraping with a headless browser; the Jotform HTML embeds all
  the visible product copy in inline text nodes already.

## Phases

### Phase 1 — Apply Jotform copy where present

Implemented by `scripts/import-jotform-descriptions.ts`. Idempotent
(running on already-imported data is a no-op aside from the timestamped
backup).

Steps:

1. **Fetch + diff.** Download the live form HTML to a temp buffer and
   byte-compare with `scripts/jotform-raw.html`. If identical, keep the
   existing file. If different, overwrite the local copy and proceed
   with the new content.
2. **Parse.** Walk the HTML for occurrences of
   `aria-label="Select Product: NAME"`. For each, grab the nearest
   following `class="form-product-description"` block (within ~8 KB of
   the same product card). Decode HTML entities, strip tags, collapse
   whitespace. Result: `{ name, text }` records, one per Jotform product.
3. **Classify.** A record is "rich" if `text.length >= 100`. Sample stats
   on the current HTML: 77 rich, 423 spec-only, 11 empty (total 511).
4. **Split rich text** into the four fields, in order:
   - `indication` = text after the first match of
     `/\bIndications?\s*:\s*/i`, up to the next section keyword or end.
   - `protocol` = text after the first match of
     `/\b(Protocol|Treatment Protocol|How to use|Directions|Application)\s*:\s*/i`,
     up to the next section keyword or end.
   - `packaging` = text after the first match of
     `/\b(Pkg|Packaging|Pack|Volume)\s*:\s*/i`, up to the next section
     keyword or end.
   - `description` = everything BEFORE the first matched section
     keyword, trimmed. If no keyword was matched at all, the entire text
     is the description.
5. **Match** Jotform name → `products.json` product using the existing
   `scripts/lib/fuzzy-match.ts` (`normalise` + `scoreMatch`),
   match-threshold 2 (same as `sync-from-jotform.ts`). Greedy
   longest-name-first to avoid the most ambiguous matches stealing
   shorter records.
6. **Apply** for every matched product whose Jotform record is rich:
   - Always overwrite `description` with the Jotform-derived value.
   - Overwrite `indication` only if the Jotform record produced one
     (otherwise keep the existing field).
   - Same rule for `protocol` and `packaging`.
   - Never overwrite a non-empty field with an empty Jotform value.
7. **Report** `scripts/import-jotform-descriptions-report.txt` with
   sections:
   - Applied rich (id, name, which of the four fields were updated).
   - Matched but spec-only (id, name, Jotform text) — this is the
     **Phase 2 candidate list**.
   - Unmatched Jotform records (name, text-length).
   - Unmatched products.json entries (id, name) — products with no
     Jotform counterpart at all.

The dev server hot-reloads `data/products.json`, so the new copy shows
in the browser immediately.

### Phase 2 — Web research for products without rich Jotform copy

**Implemented later, only after the user reviews the Phase 1 report and
explicitly approves Phase 2.**

Skeleton-only here:

- `scripts/web-research-products.ts`
- Takes a list of product IDs to process (default: every product whose
  current `description` still matches the LLM stub or whose Phase 1
  match was spec-only).
- For each product: web search by name + brand, fetch top 1-3 results,
  extract plain text, apply the same split logic as Phase 1.
- Batched (10 at a time) with a checkpoint report after each batch so
  the user can stop or revise scope.

### Phase 3 — Protocol detail enhancement

**Implemented later, only after Phase 2 is done.**

Skeleton-only:

- `scripts/refine-protocols.ts`
- Two strategies, user chooses at run time:
  - **A. Category templates.** Recommended. A small table mapping each
    of the 20 categories to a 5-15 line clinical protocol skeleton.
    Per-product substitution pulls volume/concentration from the
    product's `specification` field. Generates uniform, professional
    protocol prose without web fetches.
  - **B. Per-product web research.** Same shape as Phase 2 but targeted
    at the protocol field. Slow, expensive, but accurate.

## Verification

Phase 1:
- Script exits 0.
- `manual-skipped 0` in the report (filenames stable).
- Spot-check a known-rich product: id 187 PDRENZA SERUM & CREAM SET
  (or whichever ID corresponds) — `description` shows the Salmon PDRN /
  NMN / Pearl Tomato opening sentence; `packaging` shows the
  "1x Pdrénza Serum 50ml…" content.
- Spot-check id of MELANOSA CREAM — `description`, `indication`,
  `protocol` all populated.
- Spec-only product like MisAdi Beso (id 2) is in the Phase 2
  candidate list, not silently mutilated.
- `npx tsc --noEmit -p tsconfig.json` clean.
- Browser: open `/en/product/<rich-id>` and verify the new prose
  renders in the Description / Indication / Protocol / Packaging
  sections.

## Rollback

`copy data\backups\products-{timestamp}.json data\products.json`

The local HTML cache (`scripts/jotform-raw.html`) is unchanged unless
the live form differed, in which case the previous local copy is gone —
this is acceptable as that file is documentation, not source of truth.

## Out of scope (follow-ups)

- Phase 2 (web research) — separate spec when user is ready.
- Phase 3 (protocol refinement) — separate spec.
- Multi-language description rewrite for `data/translations/ru.json`
  and `ko.json`.
- Description / indication formatting (bullets vs prose) — the Jotform
  source uses bullets in some entries; we preserve them as plain text
  with `•` glyphs or hyphens as they appear in the HTML.
