# Product Data Fixes — Design

Date: 2026-05-19
Owner: alanhon1

## Problem

The lumiere-app catalogue has several user-visible defects rooted in product-data
extraction errors and i18n / search-state UX glitches:

1. **Broken product names** in `data/products.json`:
   - "Product NNN" placeholders where the parser failed to extract a name
     (e.g. `Product 70`, `Product 141`, `Product 203`, `Product 226`, `Product 247`,
     `Product 267`, `Product 287`, `Product 292`, `Product 309`, `Product 338`,
     `Product 386`, `Product 406`, etc.)
   - Truncated names — cut mid-word at unusual characters (em dash, subscript):
     `"C"` (id 357), `"Fere"` (id 342), `"Huto"`, `"Re N To"`, `"BM Hid"`,
     `"Muchcain"`, `"YOUTFILL DEEP"`, `"MISADI C O"`, `"DERMAGEN Lunato"`,
     `"DERMAGEN EZ C"`, `"DERMAGEN Cellsia Post"`, `"DERMAGEN Cindelria Tone - U"`,
     `"ULDM Ult"`, `"JOLLA"`, `"Sungshim"`, `"JBP"`, etc.
   - Some `specification` strings are also truncated where they should carry
     the post-em-dash subtitle (e.g. id 103 `"30 mL x 5 tubes ... matrix f"`).
   - `description` text generated by the LLM sometimes embeds the broken name,
     which then surfaces as garbage on the product page.

2. **User-supplied images** in `missing finds/` have not yet been wired into
   the catalogue. Some filenames embed the product number (`NNNproduct.webp`)
   for unambiguous mapping; others are name-only and need manual confirmation.

3. **Locale auto-detection** sends visitors with a Russian `Accept-Language`
   header to `/ru` even though the canonical entrypoint is `/en`. `proxy.ts`
   does not pass `localeDetection: false`, so next-intl falls back to header
   sniffing.

4. **Catalogue search/filter state is lost** when the user clicks into a
   product and presses *Back to Catalogue*. Only `q` is persisted to the URL;
   `activeCategory`, `sortBy`, `saleOnly`, `newOnly`, `groupedOnly`, and `page`
   live only in component state. `BackToCatalogueButton` reconstructs the
   category from `document.referrer` but discards the search/query string.

5. **Stale products** need removing:
   - All TESORO variants from Fillers (TESORO COLLAGEN in Mesotherapy stays).
   - Both HANHEAL products from Mesotherapy (`HANHEAL HA AMPOULE`, `HANHEAL PDRN BOOSTER`).
   - Seven items for which no image can be sourced and which the user no
     longer wants listed (FINASTERIDE Tab, JOLLA HIGH FREQUENCY MASSAGE CREAM,
     JOLLA PREMIUM OLIVE MASSAGE CREAM, Ferex inj, ARGININE INJ, REJUVE Inj,
     EVERLINE MEZO NEEDLE).

## Source of truth

`products.txt` (519 lines) is the canonical name/spec list. Format:

```
(N) CATEGORY NAME (#X-Y)

PRODUCT NAME — specification text
PRODUCT NAME — specification text
...
```

Category headers `(N) CATEGORY NAME (#X-Y)` tie each section to its
product-ID range. Within a section, the order of lines matches the order of
IDs in `data/products.json`.

The catalogue PDF (`APR2026- CATALOGUE.pdf`) is the final source of truth if
`products.txt` is ambiguous; in practice every name needed here is already
correct in `products.txt`.

## Non-goals

- No translation work (the broken English names are independent of the `ru` /
  `ko` translation tables in `data/translations/`).
- No category restructure; ranges stay as-is.
- No price/stock/SKU changes.
- No new schema fields on `Product`.
- No re-numbering of product IDs. Deleted IDs become gaps in the sequence;
  category counts already render dynamically.
- No changes to image filenames already correctly assigned.

## Design

### 1. Data repair (`scripts/repair-products.js`)

A new Node script, run manually (`node scripts/repair-products.js`), that
rewrites `data/products.json` in place. Idempotent — running it twice on
clean data is a no-op.

Steps in order:

1. **Backup**: copy current `data/products.json` to
   `data/backups/products.{ISO-timestamp}.json`.
2. **Parse `products.txt`** into a list of `{ categoryId, lineIndex, name, spec }`
   tuples. Category headers `(N) CATEGORY (#X-Y)` define the bucket; skip
   blank lines, headers, and lines without an em dash unless they are a
   recognised header-free entry.
3. **Build product-id → canonical-entry map** in two passes per category:
   1. **Name-similarity pass** — for each json product in the category whose
      current name is non-empty and does NOT match a broken pattern, find the
      products.txt entry in the same category whose name has the highest
      normalised prefix overlap (case- and punctuation-insensitive, after
      stripping suffixes like `(CE)`, `PLUS`, `LIDOCAINE`). Require ≥ 6
      shared characters to count as a match. Lock those matches.
   2. **Ordinal pass for the rest** — for the remaining json products
      (broken names + `Product NNN` placeholders) and the remaining
      products.txt entries in that category, pair them up in the order they
      appear. If counts disagree, abort with a per-category report; the
      operator resolves manually before re-running.

   The line/ID counts are expected to differ (products.txt has section
   headers and a small number of trailing notes); the parser strips those
   before pairing.
4. **Apply deletions (16 ids)** before anything else, so the per-category
   ordinal matches `products.txt`:
   - TESORO ids `39, 40, 41, 42, 43, 44, 45`
   - HANHEAL ids `99, 100`
   - Item 6 ids `152, 290, 292, 342, 379, 406, 411`
   (Final ID list resolved at run time by name lookup in case of off-by-one.)
5. **Rewrite name + specification** for every remaining product whose current
   `name` does not exactly equal the canonical name. The script must NOT
   overwrite already-correct names.
6. **Regenerate description** for *only* the products that just had a
   name/spec correction AND whose existing `description` contains the broken
   name as a substring (i.e. the description is poisoned). Template:

       "{name} is a professional-use product distributed for licensed
       practitioners. Specification: {specification}."

   Descriptions that are clean marketing copy stay untouched.
7. **`groupName` / `groupImage`** updates: if any group now has zero or one
   variant after deletion, clear `groupId` / `groupName` / `variantLabel` on
   the lone remaining variant so it renders as a single product (no Bundle
   badge). If a group has no remaining variants nothing to clean.
8. **Image auto-mapping**: for every file in `missing finds/` whose basename
   matches `/(\d+)\s*product/i`, copy it to
   `public/images/products/product-{id}.{ext}` (only if that target does not
   already exist OR the target is a 0-byte/empty file). Update the
   corresponding product's `image` field to the new path.
9. **Manual-mapping report**: write `scripts/repair-products.log` listing
   each `missing finds/` file without a number in its name alongside up to
   five candidate product names (Levenshtein-ranked against the filename).
   The script does not auto-apply these — the user will hand-confirm a
   mapping in a follow-up pass.
10. **Validation pass** (script exits non-zero on failure):
    - every category has ≥ 1 product
    - no surviving name matches the broken-name patterns
      (`/^Product \d+$/`, `/^[A-Za-z]{1,3}$/`, `/^[A-Z]\.?$/`)
    - 16 deletion targets are absent
    - every `groupId` has either 0 or ≥ 2 members
    - print a report: `N deletions, N name fixes, N spec fixes, N descriptions
      regenerated, N images auto-mapped, N images need manual mapping`.

### 2. Locale auto-detection off (`proxy.ts`)

Add `localeDetection: false` to the `createMiddleware` call. Effect:

- Root `/` and any unprefixed path always 302 → `/en`.
- Visitors keep whatever locale they explicitly chose via the header switcher.
- The `Accept-Language` header is ignored.

### 3. URL-driven catalogue filters

`components/catalogue/CatalogueClient.tsx`:

- Read initial state from `useSearchParams()` for: `q`, `cat`, `sale`, `new`,
  `grouped`, `sort`, `page`.
- Every filter change calls a single `updateUrl(patch)` helper that calls
  `router.replace(\`/${locale}/catalogue?${qs}\`, { scroll: false })`.
- Use `useEffect` only to subscribe to `searchParams` changes coming from
  back/forward navigation; the rest is one-way state → URL.
- Boolean filters serialize as `1`/absent; numeric ones as plain numbers;
  defaults are omitted from the URL (no `?sort=default`).

`components/catalogue/BackToCatalogueButton.tsx`:

- Parse `document.referrer`; if same-origin and pathname matches
  `/[locale]/catalogue` or `/[locale]/catalogue/[category]`, set
  `target.href = referrer.pathname + referrer.search` and label based on the
  category id if present.
- If no referrer or different origin: fallback to `/[locale]/catalogue`.

`/[locale]/catalogue/[category]/page.tsx` already delegates to
`CatalogueClient`, so it inherits the URL behavior. Verify during testing.

## Verification

Script-side:
- `node scripts/repair-products.js` exits zero with a clean report on a fresh
  checkout.
- Re-running the script is a no-op (no diff in `data/products.json`).

Manual:
- `npm run dev`, visit `http://localhost:3000/` from a browser with
  `Accept-Language: ru-RU` → lands on `/en`.
- Catalogue → search `regenovue` → click first result → *Back* → search box
  still shows `regenovue`, page is the same page number.
- Spot-check restored names: ids 70, 103, 141, 203, 226, 247, 267, 287, 292,
  309, 338, 357, 386, 406, plus a handful of the truncated ones (`Fere`,
  `Huto`, `Re N To`, `C`, `MISADI C O`, `DERMAGEN Lunato`).
- Confirm 16 deleted products are gone from the catalogue and search.
- Confirm sidebar product / category counts are sensible (438 → 422).

## Rollback

`copy data\backups\products.{timestamp}.json data\products.json`

`proxy.ts` and the two catalogue components are small, reviewable diffs; if
either is wrong, revert the commit.

## Out of scope (follow-ups)

- Resolving the manual-mapping report for unnumbered images in `missing finds/`.
- Re-translating `ru.json` / `ko.json` for renamed products.
- Refreshing per-product `enrichedInfo` (none of the affected products
  populate it today).
